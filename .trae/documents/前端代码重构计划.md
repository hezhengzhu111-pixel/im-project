## 项目概述

- **项目类型**: IM 即时通讯聊天应用

## 一、现状问题分析

### 1.1 类型系统问题
| 问题 | 描述 | 影响 |
|------|------|------|
| 类型重复 | `User` 和 `UserInfo` 有大量重复字段 | 维护困难，类型不一致 |
| 类型宽松 | `Message` 使用大量 `any` 和 `Record<string, any>` | 类型安全降低 |
| 缺少类型 | 很多 API 响应没有完整类型定义 | 编译期无法检查错误 |

### 1.2 Store 职责问题
| Store | 行数 | 问题 |
|-------|------|------|
| `useChatStore` | 1276行 | 过于庞大，职责不单一（聊天、会话、好友、群组、搜索全混在一起）|
| `useUserStore` | 663行 | 包含业务逻辑和 UI 状态混杂 |
| `useWebSocketStore` | 504行 | 消息处理逻辑复杂，与业务紧耦合 |

### 1.3 组件问题
| 组件 | 问题 |
|------|------|
| `Chat.vue` (827行) | 过长，逻辑与 UI 混杂 |
| `MessageItem.vue` (476行) | 包含业务逻辑（音频播放、右键菜单） |
| `useChatLogic.ts` | 与 `Chat.vue` 存在功能重复 |

### 1.4 代码质量问题
- **重复代码**: 多处重复实现相似功能（如时间格式化、用户搜索）
- **错误处理不一致**: 有的用 `try-catch`，有的直接 `.catch()`
- **资源泄漏**: 音频播放器未正确销毁，右键菜单事件监听未清理
- **性能问题**: 消息列表无虚拟滚动，大数据量时卡顿
- **安全隐患**: Token 存储在 localStorage，控制台日志打印敏感信息

### 阶段六：安全性与性能优化（第4周）

#### 6.1 安全加固
| 问题 | 解决方案 |
|------|----------|

| 依赖注入 | 路由懒加载、组件按需加载 |

## 五、重构原则与规范

### 5.1 代码规范
1. **单一职责**: 每个文件/函数只做一件事
2. **类型优先**: 先定义类型，再实现逻辑
3. **组合优于继承**: 使用 Composables 而非 Mixins
4. **接口隔离**: 使用细粒度的接口而非庞大对象

### 5.2 文件组织规范
```
src/
├── assets/               # 静态资源
├── components/           # 组件（按功能分组）
│   ├── common/           # 通用组件
│   ├── chat/             # 聊天相关组件
│   └── layout/           # 布局组件
├── composables/          # 组合式函数（新增）
├── config/               # 配置（精简）
├── constants/            # 常量（新增）
├── hooks/                # Hooks（与 composables 合并）
├── router/               # 路由
├── services/             # API 服务
├── stores/               # 状态管理
├── types/                # 类型定义（重构后）
├── utils/                # 工具函数
└── pages/                # 页面组件
```

### 5.3 命名规范
| 类型 | 规范 | 示例 |
|------|------|------|
| 组件 | PascalCase | `ChatContainer.vue` |
| Composables | camelCase，use 开头 | `useAudioPlayer.ts` |
| 类型/接口 | PascalCase | `UserInfo`, `ApiResponse<T>` |
| 常量 | UPPER_SNAKE_CASE | `MAX_FILE_SIZE` |
| Store | useXxxStore | `useUserStore` |
